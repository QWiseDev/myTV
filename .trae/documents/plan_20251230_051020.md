# Supabase 后端服务可行性分析报告

## 一、现有架构分析

### 1.1 存储后端切换机制

项目采用**工厂模式 + IStorage 接口抽象**实现多存储后端支持：

```
.env 配置
    ↓
NEXT_PUBLIC_STORAGE_TYPE=upstash
    ↓
src/lib/db.ts → createStorage() 工厂
    ↓
┌──────────────────────────────────────┐
│  IStorage 接口 (统一抽象层)           │
├──────────────────────────────────────┤
│  getPlayRecord()                     │
│  setPlayRecord()                     │
│  getFavorite() / setFavorite()       │
│  registerUser() / verifyUser()       │
│  ... (20+ 方法)                      │
└──────────────────────────────────────┘
    ↓
┌────────────┬────────────┬────────────┬────────────┐
│ localstorage│   redis   │  upstash   │  kvrocks   │
│  (客户端)   │ (服务端)   │ (云Redis)  │ (K-V存储)  │
└────────────┴────────────┴────────────┴────────────┘
```

**Supabase 集成策略**：只需新增 `supabase` 存储类型，实现 `IStorage` 接口即可！

### 1.2 当前数据模型

| 数据类型 | 存储 Key 模式                     | 操作频率    | 数据量级 |
| ---- | ----------------------------- | ------- | ---- |
| 播放记录 | `u:{user}:pr:{source}+{id}`   | 高       | 中等   |
| 收藏夹  | `u:{user}:fav:{source}+{id}`  | 中       | 中等   |
| 跳过配置 | `u:{user}:skip:{source}+{id}` | 低       | 小    |
| 搜索历史 | `u:{user}:sh`                 | 中       | 小    |
| 用户统计 | 缓存 key                        | 中       | 小    |
| 访问日志 | 独立 key                        | 高 (已禁用) | 大    |

### 3.2 SupabaseStorage 实现要点

```typescript
// src/lib/supabase/storage.ts
import { createClient, SupabaseClient } from '@supabase/supabase-js';
import { IStorage, PlayRecord, Favorite, AdminConfig, ... } from '@/lib/types';

export class SupabaseStorage implements IStorage {
  private client: SupabaseClient;
  
  constructor() {
    this.client = createClient(
      process.env.NEXT_PUBLIC_SUPABASE_URL!,
      process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!
    );
  }
  
  // 实现 IStorage 所有方法
  async getPlayRecord(userName: string, key: string): Promise<PlayRecord | null> {
    const [source, id] = key.split('+');
    const { data } = await this.client
      .from('play_records')
      .select('*')
      .eq('user_id', this.getUserId(userName))
      .eq('source', source)
      .eq('source_id', id)
      .single();
    return data;
  }
  
  async setPlayRecord(userName: string, key: string, record: PlayRecord): Promise<void> {
    const [source, id] = key.split('+');
    await this.client.from('play_records').upsert({
      user_id: this.getUserId(userName),
      source,
      source_id: id,
      ...record
    });
  }
  
  // ... 其他方法实现
}
```

### 3.3 数据库 Schema 设计

```sql
-- src/lib/supabase/schema.sql

-- 1. 用户配置表 (替代 profiles 配置)
CREATE TABLE admin_config (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  config JSONB NOT NULL DEFAULT '{}',
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);

-- 2. 用户表 (扩展 auth.users)
CREATE TABLE profiles (
  id UUID PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
  username TEXT UNIQUE NOT NULL,
  role TEXT DEFAULT 'user' CHECK (role IN ('owner', 'admin', 'user')),
  banned BOOLEAN DEFAULT FALSE,
  tvbox_token TEXT,
  password_hash TEXT,  -- 兼容旧系统，逐步迁移
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);

-- 3. 播放记录
CREATE TABLE play_records (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES profiles(id) ON DELETE CASCADE,
  source TEXT NOT NULL,
  source_id TEXT NOT NULL,
  title TEXT NOT NULL,
  source_name TEXT NOT NULL,
  cover TEXT,
  year TEXT,
  episode_index INTEGER NOT NULL,
  total_episodes INTEGER NOT NULL,
  original_episodes INTEGER,
  play_time INTEGER DEFAULT 0,
  total_time INTEGER DEFAULT 0,
  save_time TIMESTAMPTZ DEFAULT NOW(),
  search_title TEXT,
  remarks TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(user_id, source, source_id)
);

-- 4. 收藏夹
CREATE TABLE favorites (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES profiles(id) ON DELETE CASCADE,
  source TEXT NOT NULL,
  source_id TEXT NOT NULL,
  source_name TEXT NOT NULL,
  title TEXT NOT NULL,
  cover TEXT,
  year TEXT,
  total_episodes INTEGER NOT NULL,
  search_title TEXT,
  origin TEXT DEFAULT 'vod',
  save_time TIMESTAMPTZ DEFAULT NOW(),
  created_at TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(user_id, source, source_id)
);

-- 5. 跳过配置
CREATE TABLE episode_skip_configs (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES profiles(id) ON DELETE CASCADE,
  source TEXT NOT NULL,
  source_id TEXT NOT NULL,
  title TEXT NOT NULL,
  segments JSONB NOT NULL DEFAULT '[]',
  updated_time TIMESTAMPTZ DEFAULT NOW(),
  created_at TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(user_id, source, source_id)
);

-- 6. 搜索历史
CREATE TABLE search_history (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES profiles(id) ON DELETE CASCADE,
  keyword TEXT NOT NULL,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- RLS 策略
ALTER TABLE play_records ENABLE ROW LEVEL SECURITY;
CREATE POLICY "用户只能操作自己的播放记录" ON play_records
  FOR ALL USING (auth.uid() = user_id);

-- 索引优化
CREATE INDEX idx_play_records_user ON play_records(user_id);
CREATE INDEX idx_favorites_user ON favorites(user_id);
CREATE INDEX idx_search_history_user ON search_history(user_id);
```

## 六、优势与风险

### 6.1 引入 Supabase 的优势

| 优势       | 说明                             |
| -------- | ------------------------------ |
| **强一致性** | PostgreSQL ACID vs Redis 最终一致性 |
| **复杂查询** | 支持 JOIN、聚合等复杂操作                |
| **实时订阅** | WebSocket 实时数据同步               |
| **内置认证** | Supabase Auth 减少维护成本           |
| **成本稳定** | 固定费用 vs 按请求计费 (Upstash)        |
| **数据持久** | 完全持久化 vs 内存存储                  |

### 6.2 风险与应对

| 风险          | 可能性 | 应对措施             |
| ----------- | --- | ---------------- |
| **查询性能下降**  | 中   | 添加适当索引，使用缓存      |
| **实时功能成本**  | 低   | 限制订阅范围，按需启用      |
| **数据迁移失败**  | 中   | 完整备份，测试环境验证      |
| **API 兼容性** | 低   | 保持 IStorage 接口不变 |

## 七、最终建议

### 推荐实施路径

```
┌─────────────────────────────────────────────────────────────┐
│  方案 A: 仅作为备选存储 (推荐)                               │
├─────────────────────────────────────────────────────────────┤
│  • 继续使用 Upstash 作为主存储                              │
│  • Supabase 仅作为灾备或多活方案                            │
│  • 认证系统保持现有架构                                     │
│  • 未来按需启用 Realtime 功能                               │
│                                                              │
│  ✅ 改造量小，风险低                                        │
│  ✅ 保留现有架构稳定性                                      │
│  ⬜ 放弃部分 Supabase 优势                                  │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│  方案 B: 完整迁移方案                                       │
├─────────────────────────────────────────────────────────────┤
│  • Supabase 作为唯一存储后端                               │
│  • 迁移至 Supabase Auth                                    │
│  • 启用 Realtime 实时同步                                  │
│  • 保留 KV 兼容接口                                        │
│                                                              │
│  ✅ 充分发挥 Supabase 优势                                  │
│  ⬜ 改造量大，需要数据迁移                                  │
│  ⬜ 短期风险较高                                            │
└─────────────────────────────────────────────────────────────┘
```

**推荐选择：方案 A** (最小改造，风险可控)
